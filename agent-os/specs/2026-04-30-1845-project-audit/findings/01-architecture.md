## Notes & risks

- **§6.1.3 deviation — `ArgusServer` is not `expect/actual`.** The spec is explicit: "expect class ArgusServer is used for binding." The implementation collapses to a single `commonMain` class because Ktor's CIO engine ships KMP artifacts for JVM, Android, and Darwin. This is functionally cleaner (less platform code, no per-target binding shim) but it does not match the contract as written. Decide whether to (a) update the spec to acknowledge the multiplatform CIO path, or (b) refactor back to `expect class ArgusServer` per the contract. I recommend (a) — the existing implementation is simpler and the §3.3 intent ("only platform binding and lifecycle are platform-specific") is already met by the routing/buffer/filter code being commonMain.
- **Module inventory drift from spec §4.** Spec lists `:sample-android`. Repo ships a unified `:sample` KMP module that targets both Android and iOS (`sample/build.gradle.kts:18-70`). This is broader than the spec but not in conflict with §6.3 — the Android variant still satisfies the contract. The `:argus-okhttp` and `:argus-urlconnection` modules listed in `settings.gradle.kts:25-26` are explicitly Phase-3 deferrals per spec §7; their presence (wired only via `debugImplementation` in `:sample`) is a Phase-3 implementation, not an MVP requirement violation, but the auditor should flag this in the Phase-Boundary section (`08-phase34.md`).
- **iOS release-gate not in CI.** `verifyIosReleaseHasNoArgus` is defined and is wired into local `check`, but the GitHub workflow only runs the Android task (Linux runner). For full §6.3.7 parity with the iOS shells the workflow needs a `macos-latest` job. Low priority since the Android gate is the canonical zero-classes verification.
- **`:sample` minSdk = 34 is harness-only**, deliberately documented at `sample/build.gradle.kts:84-86` as a workaround for a D8 DEX-v40 issue with Ktor 3.2.x context parameters. The shipped `:argus-core` library still supports minSdk 24. No action needed.
- **Forbidden release-dex prefix list also blocks `Lio/ktor/server/`.** Sound — release builds must not link the embedded server. Ktor server classes leaking would otherwise produce a passing 6.3.6 check that hides a misconfigured seam.
